       - Add `'use client';` directive at top.

         - Module-level `let ctx: AudioContext | null = null`.
         - `function getContext(): AudioContext` — creates AudioContext on first call (lazy, avoids browser autoplay block).
         - `export function playOrderAlert()` — gets context, if `context.state === 'suspended'` calls `context.resume().then(() => beep(context))`, else calls `beep(context)` directly.
         - `function beep(context: AudioContext)` — creates OscillatorNode, sine wave at 880Hz (A5 — clear notification tone), gain ramp from 0.3 to 0.001 over 0.5 seconds, start and stop. This produces a short, pleasant notification beep without serving any audio files.

       - "Cancel" button: shows on RECEIVED and PREPARING status only. On click, set `showCancelConfirm = true`. Show confirmation dialog (use shadcn AlertDialog): "Cancel this order? This cannot be undone." with Cancel/Confirm buttons. On confirm, calls `onStatusUpdate(order.id, 'CANCELLED')`.
       - Style: Card component from shadcn/ui, with status-colored left border (e.g., blue for RECEIVED, orange for PREPARING, green for READY).

    5. Create `apps/web/src/components/dashboard/orders-board.tsx`:
       - `'use client'` component.
       - Props: `venueId: string`, `initialOrders: Order[]`, `token: string`.
       - State: `orders` (initialized from `initialOrders`), `connected` (boolean), `isMuted` (boolean, default false, persisted to localStorage `muted:${venueId}`), `newOrderIds` (Set<string> for highlight animation tracking).
       - Use `useEffect` with `[prop]` dependency pattern to sync `initialOrders` into state (established pattern from Phase 2 MEMORY.md).

       **WebSocket connection logic (in useEffect):**
       - Set `socket.auth = { token }`.
       - Call `socket.connect()`.
       - On `connect`: set `connected = true`, emit `'join:venue'` with venueId. Then fetch active orders via REST (`fetchActiveOrders(venueId)`) and replace local state — this handles both initial connect AND reconnect (per user decision: re-fetch on reconnect before resuming WS feed).
       - On `disconnect`: set `connected = false`.
       - On `order:new`: add order to state (`setOrders(prev => [order, ...prev])`), add order.id to `newOrderIds` set, play audio alert if `!isMuted` (call `playOrderAlert()`). Remove from newOrderIds after 2 seconds (animation duration).
       - On `order:updated`: update order in state (`setOrders(prev => prev.map(o => o.id === update.id ? { ...o, ...update } : o))`).

       **REST polling fallback:**
       - On `connect_error`: check if `!socket.active` (socket.io stopped retrying after 3 attempts). If so, start a 5-second interval calling `fetchActiveOrders(venueId)` and replacing state. This is the silent fallback per user decision.
       - Store interval ref to clear on cleanup.

       **Completed orders auto-hide:**
       - When an order's status becomes COMPLETED, start a 30-second timeout. After 30s, remove it from the `orders` array. Use a Map<orderId, timeoutId> to track pending auto-hides.
       - Clear all timeouts on unmount.

       **Cleanup:**
       - On unmount: `socket.off()` all named listeners, `socket.disconnect()`, clear polling interval, clear all auto-hide timeouts.

       **Mute button:**
       - Render a mute/unmute toggle button in the page header area. Use `Volume2` / `VolumeX` lucide icons.
       - On first click of the mute button (or any user interaction), the AudioContext will be resumed (browser autoplay policy — Pitfall 7).
       - Persist mute state to localStorage: `localStorage.setItem('muted:' + venueId, String(isMuted))`.

       **Status update handler:**
       - `handleStatusUpdate(orderId, newStatus)`: call `updateOrderStatus(venueId, orderId, newStatus)` server action. On success, optimistically update local state. The WebSocket will also emit the update, but optimistic is faster UX.

       **Layout:**
       - Render `<ConnectionBanner connected={connected} />` at the top.
       - Three columns in a CSS grid (responsive: 1 col on mobile, 3 on lg+):
         - Column "Received": filter orders where `status === 'RECEIVED'`, render OrderCard for each.
         - Column "Preparing": filter orders where `status === 'PREPARING'`.
         - Column "Ready": filter orders where `status === 'READY'`.
       - Each column has a header with the status name and a count badge.
       - CANCELLED orders are not shown (they are already excluded by the active orders endpoint, but filter client-side too for safety).

       Use `useRef` for the `isMuted` callback ref pattern (from MEMORY.md) if `isMuted` is used inside the `order:new` handler to prevent stale closure. Store the current value of `isMuted` in a ref updated each render, read from the ref inside the socket handler.

    AVOID: Creating socket inside the component (creates new connection per render — use the module-level singleton from socket.ts).
