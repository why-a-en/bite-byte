    2. Create `apps/web/src/components/dashboard/order-history-table.tsx`:
       - `'use client'` component (needs client-side date filter interaction and pagination).
       - Props: `venueId: string`, `initialData: { orders: Order[], total: number, page: number, pageSize: number }`.
       - State: `data` (initialized from initialData), `fromDate: string` (date input), `toDate: string` (date input), `currentPage: number`.

       - Date filter section: two `<input type="date">` fields for "From" and "To" dates, with a "Filter" button. On filter click (or on date change), call `fetchOrderHistory(venueId, { from: fromDate, to: toDate, page: 1 })` and update state. Use a "Clear" button to reset filters.

       - Table rendering: HTML table (or shadcn Table component if available) with columns:
         - Reference Code
         - Customer Name
         - Items (count, e.g., "3 items")
         - Total
         - Status (badge — green for COMPLETED, red for CANCELLED)
         - Date (formatted with Intl.DateTimeFormat)

       - Each row is expandable (click to show item details) — similar pattern to order-card but simpler (no action buttons, just info).

       - Pagination: "Previous" / "Next" buttons with page indicator ("Page 1 of 5"). On click, fetch the new page via `fetchOrderHistory(venueId, { from, to, page: newPage })`.

       - Use `fetchApi` import from `@/lib/api` for client-side fetching (the action file wraps fetchApi already, but the client component needs to call it — use the action directly since it reads cookies server-side). Actually, since this is a client component that needs to re-fetch with different params, use a direct fetch call to the API with the access_token cookie (the browser will send cookies automatically with credentials: 'include'). Create a `fetchHistoryClient` helper within the component or a separate client-side fetch utility.

       IMPORTANT: For client-side data fetching in this component, make direct fetch calls to the NestJS API (process.env.NEXT_PUBLIC_API_URL) with `credentials: 'include'` to forward the JWT cookie. Do NOT use server actions for client-side pagination/filtering — server actions require form submission, not imperative data fetching. Import `fetchApi` pattern does not work from client components (it reads cookies server-side).

       Alternative approach: Use `useRouter().push()` with query params to trigger a full Server Component re-render with the new filters. This is simpler and keeps data fetching on the server. The history page Server Component would read `searchParams` for from/to/page and pass them to fetchOrderHistory.

       RECOMMENDED: Use the searchParams approach — the history page reads `searchParams.from`, `searchParams.to`, `searchParams.page` and fetches server-side. The client component only handles the UI for selecting dates and navigating pages by updating the URL. This avoids client-side API calls entirely and is the Next.js App Router pattern.
