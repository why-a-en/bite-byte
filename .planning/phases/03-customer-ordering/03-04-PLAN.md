**Step 1 — Create usePolling hook:**

```typescript
// apps/web/src/lib/use-polling.ts
'use client';
import { useEffect } from 'react';
import { useRouter } from 'next/navigation';

/**
 * Polls by calling router.refresh() on an interval.
 * router.refresh() triggers Server Component re-render with fresh data from the API.
 * This is the Next.js App Router native approach — no external data library needed.
 *
 * NOTE: router from useRouter() is stable in Next.js App Router — safe to omit from deps.
 * ms changes are rare — also omitted to match the canonical usePolling pattern.
 */
export function usePolling(ms: number, enabled = true) {
  const router = useRouter();

  useEffect(() => {
    if (!enabled) return;
    const id = setInterval(() => router.refresh(), ms);
    return () => clearInterval(id);
  }, [enabled]); // eslint-disable-line react-hooks/exhaustive-deps
}
```

**Step 2 — Create order status page Server Component:**

The page fetches the order status on every render (triggered by router.refresh() from the polling hook).

```typescript
// apps/web/src/app/(menu)/menu/[slug]/order/[orderId]/page.tsx
import { notFound } from 'next/navigation';
import { fetchPublicApi } from '@/lib/api-public';
import { OrderStatus } from '@/components/ordering/order-status';

export interface PublicOrder {
  id: string;
  referenceCode: string;
  status: 'PENDING_PAYMENT' | 'RECEIVED' | 'PREPARING' | 'READY' | 'COMPLETED' | 'CANCELLED';
  customerName: string;
  totalAmount: string;
  createdAt: string;
  items: Array<{
    itemNameAtOrder: string;
    unitPriceAtOrder: string;
    quantity: number;
  }>;
}

interface PageProps {
  params: Promise<{ slug: string; orderId: string }>;
}

export default async function OrderStatusPage({ params }: PageProps) {
  const { slug, orderId } = await params;

  let order: PublicOrder;
  try {
    order = await fetchPublicApi(`/public/venues/${slug}/orders/${orderId}`);
  } catch {
    notFound();  // Catches both 404 (not found) and 403 (wrong slug) — both become 404 to prevent snooping
  }

  return (
    <div className="min-h-screen bg-white">
      <div className="max-w-lg mx-auto px-4 py-6">
        <h1 className="text-xl font-bold mb-1">Order Status</h1>
        <p className="text-gray-500 text-sm mb-6">
          We'll update this page as your order progresses.
        </p>
        <OrderStatus order={order} venueSlug={slug} />
      </div>
    </div>
  );
}
```

Note: `fetchPublicApi` is called server-side on EVERY request (no caching). Next.js fetch cache is bypassed by default in Server Actions context — but for route handlers/pages, add `{ cache: 'no-store' }` to ensure fresh data on every router.refresh():

```typescript
order = await fetchPublicApi(`/public/venues/${slug}/orders/${orderId}`, { cache: 'no-store' });
```

**Render structure:**

2. Customer name: `${order.customerName}'s order`
3. If status === 'PENDING_PAYMENT': show "Awaiting payment confirmation..." with a spinner/message

**localStorage recovery for returning customers (ORDR-10):**

The order ID is already in the URL (customer navigates to `/menu/[slug]/order/[orderId]`). The `lastOrder:[venueSlug]` localStorage key is written by checkout-form.tsx when the order is created. This plan does NOT need to read localStorage — the URL already contains the orderId. The localStorage key exists as a recovery mechanism: if the customer closes the browser and reopens the venue menu page, the menu page (or a "Return to Order" banner) can read `lastOrder:[venueSlug]` to link them back.

Add a "Return to menu" link at the bottom: `← Back to Menu` linking to `/menu/${venueSlug}`.

**IMPORTANT:** This component is `'use client'` ONLY because it calls `usePolling`. It receives `order` as a prop from the Server Component parent. The Server Component refetches on every `router.refresh()` call from the polling interval — this is the Next.js App Router data update pattern.
