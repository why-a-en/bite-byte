    **Step 2: Initialize shadcn/ui**

    CRITICAL: Run from within apps/web directory (Pitfall 7 from RESEARCH.md — shadcn CLI detects framework from local package.json):
    ```bash
    cd apps/web && pnpm dlx shadcn@latest init
    ```

    When prompted: use default style, set components path to `@/components`, set utils path to `@/lib/utils`. If non-interactive mode available, use `--defaults`.

    Install required shadcn components:
    ```bash
    cd apps/web && pnpm dlx shadcn@latest add button input label card
    ```

    The `form` component for react-hook-form integration:
    ```bash
    cd apps/web && pnpm dlx shadcn@latest add form
    ```

    This installs react-hook-form and @hookform/resolvers as peer deps.

    **Step 3: Install jose for Edge-compatible JWT verification**

    ```bash
    pnpm add --filter @bite-byte/web jose
    ```

    **Step 4: Create Next.js middleware**

    Create `apps/web/src/middleware.ts`:
    - Import `jwtVerify` from `jose` (NOT jsonwebtoken — Pitfall 1 from RESEARCH.md: Edge Runtime incompatibility)
    - Read `access_token` from cookies: `req.cookies.get('access_token')?.value`
    - If no token, redirect to `/login`
    - If token exists, verify with `jwtVerify(token, new TextEncoder().encode(process.env.JWT_SECRET))`
    - If verification fails (expired, invalid), clear cookie and redirect to `/login`
    - If valid, call `NextResponse.next()`
    - Export config.matcher: `['/dashboard/:path*', '/venues/:path*']` — only protect dashboard routes

    **Step 5: Configure API_URL environment variable**

    Add `API_URL=http://localhost:3001/api` to `apps/web/.env.local` (or `.env`). This is used by server actions to call the NestJS API. Add to `.env.example` as well.

    Also add `JWT_SECRET` to the web app's env (same value as API) — needed by middleware for JWT verification.

  <action>
    **Step 1: Create auth server actions**

    Create `apps/web/src/actions/auth.ts` with 'use server' directive:

    `registerAction(prevState, formData)`:
    - Extract email and password from formData
    - Validate with Zod (email format, password min 8 chars)
    - POST to `${process.env.API_URL}/auth/register` with { email, password }
    - If 409 (email exists), return { error: 'Email already registered' }
    - If success, extract access_token from response
    - Set httpOnly cookie: `cookies().set('access_token', access_token, { httpOnly: true, secure: process.env.NODE_ENV === 'production', sameSite: 'lax', maxAge: 60 * 60 * 24 * 7, path: '/' })`
    - redirect('/dashboard')

    `loginAction(prevState, formData)`:
    - Same pattern: extract, validate, POST to /auth/login, set cookie, redirect

    `logoutAction()`:
    - `cookies().delete('access_token')`
    - redirect('/login')

    Use the `useActionState` (React 19) pattern for form actions — returns [state, formAction, isPending].

    **Step 2: Create auth route group layout**

    Create `apps/web/src/app/(auth)/layout.tsx`:
    - Simple centered layout (no sidebar) — clean auth pages
    - Centered card container with max-w-md, vertically centered
    - Include project name/logo at top

    **Step 3: Create register page**

    Create `apps/web/src/app/(auth)/register/page.tsx`:
    - Use shadcn Card component wrapping the form
    - Email input (type="email", required)
    - Password input (type="password", required, min 8)
    - Submit button with loading state (isPending from useActionState)
    - Display error message from server action state
    - Link to /login ("Already have an account? Log in")
    - Use react-hook-form with Zod resolver for client-side validation
    - Form action submits via the registerAction server action

    **Step 4: Create login page**

    Create `apps/web/src/app/(auth)/login/page.tsx`:
    - Same layout as register but for login
    - Email + password fields
    - Submit button with loading state
    - Error message display
    - Link to /register ("Don't have an account? Sign up")
    - Use react-hook-form with Zod resolver
    - Form action submits via the loginAction server action

    Design priorities (from CONTEXT.md):
    - Clean and functional — simple forms, no clutter
    - Mobile-responsive — card layout works on all screens
    - Fast to learn — obvious form fields, clear CTAs
