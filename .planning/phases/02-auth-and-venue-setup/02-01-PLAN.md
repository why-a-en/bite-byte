    Add a User model to apps/api/prisma/schema.prisma:

    ```prisma
    model User {
      id           String   @id @default(uuid()) @db.Uuid
      email        String   @unique
      passwordHash String   @map("password_hash")
      createdAt    DateTime @default(now()) @map("created_at")
      updatedAt    DateTime @updatedAt @map("updated_at")

      venues Venue[]

      @@map("users")
    }
    ```

    Add ownerId to the existing Venue model:
    - Add `ownerId String @map("owner_id") @db.Uuid` field
    - Add `owner User @relation(fields: [ownerId], references: [id])` relation

    The `users` table does NOT get RLS policies (users are not tenant-scoped; they own venues, they don't belong to venues). The ownerId on Venue is scoped at the application layer via `WHERE ownerId = req.user.userId`.

    Run migration via the Docker pattern established in Phase 1 (WSL2 SCRAM-SHA-256 proxy constraint — must run inside Docker container on bite-byte_default network). Use superuser (bitebyte) for migration:

    ```bash
    docker run --rm --network bite-byte_default \
      -v $(pwd)/apps/api/prisma:/app/prisma \
      -v $(pwd)/apps/api/prisma.config.ts:/app/prisma.config.ts \
      -w /app \
      -e DATABASE_URL=postgresql://bitebyte:bitebyte@bite-byte-postgres-1/bitebyte_dev \
      node:22-alpine sh -c "npm install -g prisma@7 && prisma migrate dev --name add_user_and_venue_owner"
    ```

    If the Docker migration pattern fails (known WSL2 issue), fall back to: generate SQL with `prisma migrate diff --from-migrations-directory --to-schema-datamodel`, create migration directory manually, apply via `docker compose exec postgres psql`.

    After migration, regenerate Prisma client:
    ```bash
    cd apps/api && npx prisma generate
    ```

    CRITICAL: The existing Venue table has data (test rows from Phase 1 integration tests). The migration must handle the NOT NULL ownerId column on existing rows. Use a default UUID or make ownerId nullable initially and backfill — OR if the dev database can be reset, that is acceptable for dev environment. The simplest approach: make ownerId required but provide a SQL default in the migration that creates a placeholder user for existing venue rows, then remove the default.

  <action>
    Install auth dependencies:
    ```bash
    pnpm add --filter @bite-byte/api @nestjs/passport @nestjs/jwt passport passport-local passport-jwt bcrypt
    pnpm add --filter @bite-byte/api -D @types/bcrypt @types/passport-local @types/passport-jwt
    ```

    Create UsersModule and UsersService:
    - `users.service.ts`: inject PrismaService, implement findByEmail(email), findById(id), create(email, passwordHash). Always use Prisma `select` to exclude passwordHash in find methods (return SafeUser type = Omit<User, 'passwordHash'>). The create method returns the full user for internal use by AuthService.
    - `users.module.ts`: exports UsersService so AuthModule can import it.

    Create AuthModule:
    - `auth.service.ts`:
      - `register(email, password)`: hash password with bcrypt (12 rounds), call usersService.create, sign JWT with { sub: user.id, email: user.email }, return { access_token }. Throw ConflictException if email exists.
      - `validateUser(email, password)`: find user by email (with passwordHash), bcrypt.compare, return SafeUser or null.
      - `login(user)`: sign JWT payload { sub: user.id, email: user.email }, return { access_token }.

    - `auth.controller.ts`:
      - `POST /auth/register`: accepts { email, password } body, calls authService.register, returns { access_token }.
      - `POST /auth/login`: uses @UseGuards(LocalAuthGuard), req.user is populated by Passport, calls authService.login(req.user), returns { access_token }.

    - `strategies/local.strategy.ts`: extends PassportStrategy(Strategy) from passport-local. CRITICAL: pass `{ usernameField: 'email' }` to super() (Pitfall 2 from RESEARCH.md). validate() calls authService.validateUser.

    - `strategies/jwt.strategy.ts`: extends PassportStrategy(Strategy) from passport-jwt. Extract from Bearer header via ExtractJwt.fromAuthHeaderAsBearerToken(). Secret from process.env.JWT_SECRET. validate(payload) returns { userId: payload.sub, email: payload.email }.

    - `guards/local-auth.guard.ts`: extends AuthGuard('local').
    - `guards/jwt-auth.guard.ts`: extends AuthGuard('jwt'). This guard is exported and used by all protected endpoints in Plans 02+.

    - `auth.module.ts`: imports UsersModule, PassportModule, JwtModule.register({ global: true, secret: process.env.JWT_SECRET, signOptions: { expiresIn: '7d' } }). Providers: AuthService, LocalStrategy, JwtStrategy. Controllers: AuthController. Exports: AuthService.

    Update AppModule: add AuthModule and UsersModule to imports array.

    Add JWT_SECRET to .env.example and apps/api/.env (use a random 32+ char string for dev).
