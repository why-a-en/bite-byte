## Decisions Made
- **Prisma 7 config-file architecture:** Prisma 7 removed `url = env(...)` from `datasource db {}` in schema.prisma. The URL now lives exclusively in `prisma.config.ts`. This is a breaking change from all pre-v7 Prisma docs and from the RESEARCH.md patterns — was caught immediately by the schema validator.
- **Docker port 5433:** Local PostgreSQL 14 was already listening on `127.0.0.1:5432`. Changed Docker Compose port mapping to `5433:5432` to avoid the conflict.
- **Migrations via Docker container:** Even on port 5433, Prisma CLI on the WSL2 host couldn't authenticate through the Docker Desktop Windows proxy (the connection was intercepted before reaching the container). Ran `prisma migrate dev` inside a `node:22-alpine` container on the `bite-byte_default` Docker network, using the container hostname `bite-byte-postgres-1` for direct network access.
- **Migration file reconstruction:** The first successful migration run was in an ephemeral container (no output volume). Used `prisma migrate diff --from-empty --to-schema` to regenerate the exact SQL, created the migration directory locally, and verified with `prisma migrate status` inside the container showing "Database schema is up to date!"

## Deviations from Plan

### Auto-fixed Issues

**1. [Rule 3 - Blocking] Prisma 7 removed datasource url from schema.prisma**
- **Found during:** Task 1 (schema validation)
- **Issue:** Plan and RESEARCH.md Pattern 4 included `url = env("DATABASE_URL")` in `datasource db {}` — invalid in Prisma 7.4.2. Error: "The datasource property `url` is no longer supported in schema files."
- **Fix:** Removed `url` property from `datasource db` block in schema.prisma. URL is defined in prisma.config.ts `datasource.url` (which prisma init generates automatically).
- **Files modified:** apps/api/prisma/schema.prisma, apps/api/prisma.config.ts
- **Verification:** `npx prisma validate` shows "The schema at prisma/schema.prisma is valid"
- **Committed in:** 5991a21 (Task 1 commit)

**2. [Rule 3 - Blocking] Local PostgreSQL 14 occupying port 5432 on WSL2 host**
- **Found during:** Task 2 (prisma migrate dev)
- **Issue:** `/etc/postgresql/14/main` PostgreSQL instance was bound to `127.0.0.1:5432`, intercepting all connections before Docker container. Prisma P1000 auth error — bitebyte user doesn't exist in local postgres.
- **Fix:** Changed docker-compose.yml port from `5432:5432` to `5433:5432`. Updated DATABASE_URL in .env and .env.example to use port 5433.
- **Files modified:** docker-compose.yml, apps/api/.env, .env.example
- **Verification:** Docker container healthy on port 5433; `nc -zv 127.0.0.1 5433` succeeds
- **Committed in:** 2c0f508 (Task 2 commit)

**3. [Rule 3 - Blocking] WSL2 Docker Desktop proxy auth failure on port 5433**
- **Found during:** Task 2 (prisma migrate dev)
- **Issue:** Even after switching to port 5433, Prisma CLI and psql on the WSL2 host received P1000 auth errors. Connection never appeared in Docker postgres logs — Windows-side Docker Desktop proxy intercepts and fails SCRAM-SHA-256 auth before reaching the container. This is a known WSL2/Docker Desktop networking limitation.
- **Fix:** Ran `prisma migrate dev --name init` inside a `node:22-alpine` container using `--network bite-byte_default` and `DATABASE_URL=postgresql://bitebyte:bitebyte@bite-byte-postgres-1/bitebyte_dev` (container-to-container DNS, no port proxy involved). Migration applied successfully.
- **Migration file handling:** The ephemeral container's migration files weren't on the host. Used `prisma migrate diff --from-empty --to-schema` to regenerate identical SQL, created `apps/api/prisma/migrations/20260302173804_init/migration.sql` on host, verified with `prisma migrate status` inside container.
- **Files modified:** apps/api/prisma.config.ts (graceful dotenv), apps/api/prisma/migrations/ (created)
- **Committed in:** 2c0f508 (Task 2 commit)

## Issues Encountered
- Prisma 7 schema format changed from Prisma 5/6 — datasource URL no longer in schema file. RESEARCH.md Pattern 4 was written for Prisma 5/6 syntax. Caught immediately by `prisma validate`.
- WSL2 Docker Desktop port proxy doesn't forward SCRAM-SHA-256 authentication correctly for ports bound to 0.0.0.0 by Docker containers. The postgres container's `pg_hba.conf` uses scram-sha-256 for non-loopback, which the Windows proxy fails to pass through. Using container-to-container networking (same Docker network) bypasses this entirely.

## User Setup Required
None for the migration itself. However, note:
- `docker compose up -d postgres` starts the container on port 5433
- `DATABASE_URL="postgresql://bitebyte:bitebyte@localhost:5433/bitebyte_dev"` in apps/api/.env
- Future `prisma migrate dev` runs on this machine must be run inside a Docker container due to the WSL2 proxy auth issue. Use: `docker run --rm --network bite-byte_default -v $(pwd)/prisma:/app/prisma -v $(pwd)/prisma.config.ts:/app/prisma.config.ts -w /app -e DATABASE_URL=postgresql://bitebyte:bitebyte@bite-byte-postgres-1/bitebyte_dev node:22-alpine sh -c "npm install -g prisma@7 && prisma migrate dev --name <name>"`
